# セキュリティ実装ガイド（Security Implementation）

## このファイルの目的

セキュリティの知識がなくてもAIと協働して安全な実装ができるための手引き。
「何を実装するか」ではなく「AIにどう問うか」を提供する。

チェックリスト（production-readiness.md）が「確認する」ためのものなら、
このファイルは「実装する」ためのもの。

---

## 原則：AIへの問い方が品質を決める

```
悪い問い：「ログインを実装して」
→ AIは「動くコード」を生成するが、セキュリティ要件は未定義のまま

良い問い：「以下の要件でログインを実装して。
           - パスワードはbcryptでハッシュ化（ラウンド数12以上）
           - JWTの有効期限は1時間・リフレッシュトークンは7日
           - ログイン失敗5回でアカウントロック（15分）
           - 失敗ログをERRORレベルで記録（パスワードは含めない）」

知識がない場合の正しい問い：
「ログインを実装したい。
 このプロジェクトの規模と目的（ARCHITECTURE.md参照）に対して、
 セキュリティ上の要件として何を決めてから実装すべきか教えて。
 決まったら実装して。」
→ AIが要件を引き出してから実装する。知識がなくても機能する。
```

**「分からないまま実装する」と「分からないことをAIに聞いてから実装する」の差が、
6ヶ月後のコードベースの安全性を決定する。**

## 実装パターン別：AIへの指示テンプレート

### パスワード認証

```
パスワードの保存・検証を実装して。

要件：
- bcryptまたはArgon2でハッシュ化（平文は一切保存しない）
- ハッシュ化のコスト係数（bcryptならラウンド数12）を設定する
- パスワードはログにも出力しない（デバッグログも含む）
- レートリミット（5回失敗で15分ロック）はミドルウェアで実装する
- エラーメッセージは「メールアドレスまたはパスワードが違います」に統一する
  （どちらが間違いか教えない）
```

### JWT認証

```
JWTによる認証を実装して。

要件：
- アクセストークン有効期限：15〜60分（短期・リソースアクセスに使う）
- リフレッシュトークン有効期限：7〜30日（DBに保存・ローテーション運用）
- アルゴリズムはHS256またはRS256を明示指定（alg:none攻撃対策）
- トークンの保存場所：HttpOnly Cookie（localStorageは禁止・XSS脆弱性）
- リフレッシュトークンはローテーション：使用時に古いトークンを無効化し新しいものを発行
- ログアウト時はDBのリフレッシュトークンを削除する
```

### 入力バリデーション

```
以下のエンドポイントに入力バリデーションを追加して。

要件：
- バリデーションはバックエンドで必ず実施（フロントは補助のみ・信頼しない）
- zodまたはjoi等のスキーマバリデーションライブラリを使う
- バリデーションエラーはユーザーに分かる形で返す（内部情報は含めない）
- 数値は範囲チェック・文字列は長さ上限・型チェックを含める
- SQLクエリはORMのパラメータバインディングのみ使用（文字列連結禁止）
```

### エラーレスポンス

```
エラーハンドリングを実装して。

要件：
- ユーザーへのレスポンスにスタックトレース・内部パス・DB情報を含めない
- 本番環境では「予期しないエラーが発生しました」に統一する
- 内部エラーはERRORレベルでログに記録（スタックトレース含む）
- NODE_ENV === 'production' の場合に詳細を隠す分岐を入れる
- エラーコードは共通の AppError クラスを使う（shared/errors/に定義）
```

### APIキー・シークレット管理

```
外部APIとの通信を実装して。

要件：
- APIキーは環境変数から読み込む（コードにハードコードしない）
- 環境変数の読み込みは config/env.ts で一元管理する（zodでバリデーション付き）
- APIキーはログに出力しない（エラーログにも含めない）
- APIキーをURLパラメータに含めない（ヘッダーで送信する）
- .env.example に変数名だけ追記する（値は書かない）
```

### ログ設計

```
ログ出力の設計を実装して。

要件：
- ログレベル：ERROR（即時対応）/ WARN（注意）/ INFO（通常）/ DEBUG（開発のみ）
- 本番ではDEBUGログを出力しない（NODE_ENVで制御）
- 必ず含める情報：タイムスタンプ（ISO 8601）・ログレベル・処理内容・リクエストID
- 禁止事項：パスワード・APIキー・個人情報をログに出力しない
- 構造化ログ（JSON形式）で出力する（検索・分析が容易になる）
```

## 実装後のセキュリティ確認フロー

実装が完了したら毎回このフローを実行する。

```
Step 1：自動チェック
  npm audit / pip-audit で既知の脆弱性を確認

Step 2：@security-auditor に依頼
  「[ファイルパス]を監査してください。
   特に以下を重点的に確認してください：
   - 機密情報（パスワード・APIキー）の扱い
   - 外部入力のバリデーション
   - エラーレスポンスへの内部情報混入
   - 認証が必要な処理への保護漏れ」

Step 3：人間が確認する
  AIが「問題なし」と報告しても、以下だけ自分で確認する：
  - .env ファイルがコミットされていないか（git status で確認）
  - ハードコードされたパスワード・APIキーがないか（git diff で確認）
```

## 「知らないこと」への対処法

実装中に「これは安全か分からない」と感じたとき：

```
1. 実装を止める

2. AIに問う：
   「この実装にセキュリティ上のリスクはあるか。
    あれば具体的に何が危険かと、安全な代替実装を教えて」

3. 理解した上で採用する：
   「なぜ元の実装が危険で、新しい実装が安全なのか日本語で説明して」
   → 説明を理解してから採用する。理解せずに使わない。

4. decisions/ に記録する：
   「このセキュリティ判断（何を・なぜ・どうした）をdecisions/に記録して」
   → 将来の自分への説明になる
```

## 多層防御の実装チェックリスト

「防ぎ切れなかった場合」の備えを実装する。
Layer2〜4（検知・限定・回復）の詳細チェックリストは `principles/resilience.md` を参照。

セキュリティ実装で特に確認すること（Layer1：侵入を防ぐ）：
```
[ ] 認証・認可が実装されているか
[ ] 入力バリデーションがバックエンドで実施されているか
[ ] 通信がHTTPSのみか
[ ] エラーログとアラートが機能しているか（レートリミット含む）
```

## AIへの活用方法

このファイルをセッション冒頭に渡すことで、AIが安全な実装を標準として生成する。

```
principles/security-implementation.md を読んでから実装を始めてください。
今回実装するのは [機能名] です。
```
