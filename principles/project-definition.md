# プロジェクト定義（Project Definition）

開発を始める前に決めること。
技術選定・アーキテクチャ・ディレクトリ構成はすべてここから導かれる。

---

## なぜ先に定義するのか

「何を作るか」が曖昧なまま実装を始めると：
- 技術選定が後から覆る（SSRが不要だったのにNext.jsを選んでいた、など）
- 設計の前提が崩れ、大規模な作り直しが発生する
- AIへの指示が毎回ブレる（文脈が定まっていないため）

AIと開発するとき、この定義ファイルを渡すことで「なぜこのプロジェクトが存在するか」を毎回説明せずに済む。

---

## 定義すべき5項目

### 1. 目的（Why）

このプロジェクトが存在する理由。解決する問題。
「〜できるようにするため」という形で書く。

```
悪い例：「株式分析ツールを作る」（何をするかしか書いていない）
良い例：「ファンダメンタル分析に基づく投資判断を、複数の情報源を行き来せずに
         1画面で完結させるため」（なぜ作るかが明確）
```

### 2. 対象ユーザー（Who）

誰が使うか。自分だけか・チームか・一般公開か。
ユーザーの技術レベル・使用環境（PC/スマホ）・利用頻度も書く。

これによって以下が決まる：
- UIの複雑さ（自分だけなら粗くていい・公開するなら丁寧に）
- 認証の必要性（自分だけなら不要なことが多い）
- パフォーマンス要件（1人なら低スペックでOK）

### 3. 機能要件（What）

何ができるか。MoSCoW法で優先度を整理する。

```
Must（必須）：これがなければ成立しない
  - 株式の基本情報を表示する
  - スクリーニング条件を設定して銘柄を絞り込む

Should（重要だが必須ではない）：
  - スクリーニング条件を保存する

Could（あれば良い）：
  - 銘柄同士を比較する

Won't（今回はやらない）：
  - リアルタイムの株価更新
```

「Won't」を明示することがAI協働で特に重要。
未定義だとAIが過剰実装する（キャッシュ・リアルタイム更新・認証を勝手に追加するなど）。

### 4. 技術制約（Constraint）

選択の自由度を制限する固定条件。

```
言語の制約：
  - すでに習熟している言語がある場合
  - チームの技術スタックに合わせる必要がある場合

環境の制約：
  - デプロイ先が決まっている（GitHub Pages・Vercel・AWS等）
  - 使用できる外部APIが限られている

コストの制約：
  - 月額コストの上限
  - 無料枠内で収める

ライセンスの制約：
  - 商用利用するか否か（MITライセンスのみ使えるか等）
```

### 5. 成功基準（Definition of Done）

「完成」をどう定義するか。数値で表せるものは数値で書く。

```
悪い例：「使いやすいツールが完成したら」
良い例：
  - Must要件がすべて動作する
  - 主要操作が3クリック以内で完了する
  - 自分が毎日実際に使っている状態になっている
```

---

## AIへの活用方法

このファイルをセッション冒頭に渡すことで、AIが「なぜこれを作っているか」を理解した上でコードを生成する。

特に **Won't** セクションが重要。
「リアルタイム更新は今回やらない」と明示すれば、AIはWebSocketやポーリングの実装を提案しなくなる。

また**成功基準**を渡すことで、「これはMust要件を満たすか」という観点でAIにレビューを依頼できる。
