# 本番運用を想定した設計チェックリスト（Production Readiness）

## このファイルの目的

「動く」と「本番で運用できる」は別物。
このチェックリストは、プロトタイプから本番リリースに移行するときに確認するもの。

開発中から意識しておくことで、リリース直前の大規模な作り直しを防ぐ。

---

## チェックリストの使い方

リリース前にこのリストを確認する。
「未対応」のまま進める場合は、理由をADRに記録する（意図的な妥協と技術的負債の区別のため）。

---

## 1. セキュリティ

### 認証・認可
```
[ ] 認証が必要なエンドポイントはすべて保護されているか
[ ] 認証トークンの有効期限が設定されているか
[ ] ロールベースのアクセス制御が正しく機能しているか
[ ] 管理者機能へのアクセスが一般ユーザーに制限されているか
```

### データ保護
```
[ ] パスワードがハッシュ化されているか（平文保存は絶対禁止）
[ ] APIキー・シークレットが環境変数で管理されているか
[ ] コードリポジトリに機密情報がコミットされていないか（git-secretsなどで検証）
[ ] 本番DBへのアクセスが最小権限になっているか
[ ] 通信がHTTPS（TLS）で暗号化されているか
```

### 入力値・出力値
```
[ ] すべての外部入力がバックエンドでバリデーションされているか
[ ] エラーレスポンスにスタックトレースや内部情報が含まれていないか
[ ] SQLインジェクション対策がされているか（ORMのパラメータバインディングを使用）
[ ] XSS対策がされているか（ユーザー入力をエスケープせずにHTMLに出力していない）
[ ] CSRFトークンが実装されているか（フォーム送信がある場合）
```

### 依存ライブラリ
```
[ ] npm audit / pip audit でHIGH以上の脆弱性がないか
[ ] 使用していないパッケージが削除されているか
[ ] ライブラリのバージョンが固定されているか（package-lock.json / requirements.txt）
```

---

## 2. 信頼性・可用性

### エラーハンドリング
```
[ ] 想定外のエラーがユーザーに生のエラーメッセージを返していないか
[ ] すべての例外がキャッチされ、ログに記録されているか
[ ] 外部APIの呼び出しにタイムアウトが設定されているか
[ ] 外部APIの失敗時にリトライロジックがあるか（必要な場合）
[ ] DB接続失敗時のフォールバック処理があるか（必要な場合）
```

### データ整合性
```
[ ] 重要な操作がトランザクションで保護されているか
[ ] DBのユニーク制約・NOT NULL制約が適切に設定されているか
[ ] データの二重送信を防ぐ仕組みがあるか（冪等性の確保）
```

### バックアップ
```
[ ] DBのバックアップが設定されているか
[ ] バックアップからの復元手順が文書化されているか
[ ] バックアップが実際に復元できることを確認したか
```

---

## 3. パフォーマンス

### フロントエンド
```
[ ] Core Web Vitals（LCP・FID・CLS）が目標値を満たしているか
[ ] 画像が最適化されているか（WebP形式・適切なサイズ）
[ ] 不要なJavaScriptが本番バンドルに含まれていないか
[ ] コード分割（Code Splitting）が適用されているか（大規模の場合）
[ ] 静的アセットにキャッシュヘッダーが設定されているか
```

### バックエンド・DB
```
[ ] 頻繁に使うクエリにインデックスが設定されているか
[ ] N+1問題が発生していないか（ORMのクエリログで確認）
[ ] 重いクエリにページネーションが実装されているか
[ ] 必要に応じてキャッシュが実装されているか
[ ] 本番相当のデータ量でパフォーマンスを検証したか
```

---

## 4. 監視・ログ

### ログ設計
```
[ ] ログレベルが適切に設定されているか（本番でDEBUGログが出ていないか）
[ ] ログにパスワード・APIキーが含まれていないか
[ ] エラーログに十分な情報（タイムスタンプ・コンテキスト・スタックトレース）があるか
[ ] ログの保存先・保存期間が決まっているか
```

### アラート・通知
```
[ ] エラーが発生したときに通知が来るか（メール・Slack等）
[ ] インフラのコスト上限アラートが設定されているか（クラウド利用の場合）
[ ] サービスのダウンを検知する仕組みがあるか
```

---

## 5. デプロイ・運用

### 環境管理
```
[ ] 開発・ステージング・本番の環境が分離されているか
[ ] 環境ごとに異なる設定が環境変数で管理されているか（コードで分岐していないか）
[ ] 本番DBに開発用データが入っていないか
```

### デプロイプロセス
```
[ ] デプロイ手順が文書化されているか（README or docs/）
[ ] デプロイに手動の秘伝のコマンドが必要な状態になっていないか
[ ] ロールバック手順が決まっているか（デプロイ後に問題が発覚した場合）
[ ] CIでテストが自動実行されているか（テストが通らないとデプロイできない）
```

### ドキュメント
```
[ ] セットアップ手順がREADMEに書かれているか（新メンバーが1人で環境構築できるか）
[ ] .env.exampleが最新の状態か
[ ] APIのエンドポイント一覧が文書化されているか
[ ] 既知の問題・制限事項が記録されているか
```

---

## 6. テスト

```
[ ] ビジネスロジックの重要な部分にユニットテストがあるか
[ ] 主要なAPIエンドポイントにインテグレーションテストがあるか
[ ] テストが本番コードと同じリポジトリで管理されているか
[ ] テストが自動実行されているか（CI/CD）
[ ] テストカバレッジの目標が定義されているか（例：重要なロジックは80%以上）
```

---

## リリース判断の基準

### 必須（これがないとリリースしない）
- セキュリティ：認証・HTTPSは必須
- セキュリティ：機密情報の平文保存・コード直書きは禁止
- エラーハンドリング：ユーザーにスタックトレースを見せない
- バックアップ：本番DBのバックアップ設定

### 推奨（可能であれば対応する）
- パフォーマンス：インデックス・ページネーション
- 監視：エラー通知

### 許容可能な妥協（理由をADRに記録する）
- テストカバレッジが低い（理由：プロトタイプ段階）
- 監視が手動確認のみ（理由：ユーザー数が少ない）
- スケーリング設計がない（理由：当面は小規模のため）

## AIへの活用方法

このチェックリストをAIに渡すことで、コードレビューを依頼できる。

```
以下のチェックリスト（production-readiness.md）に基づいて、
このコードが本番運用に耐えられるか評価してください。
未対応の項目と、対応が必要な理由を指摘してください。

[コードを貼り付け]
```
